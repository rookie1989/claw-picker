## ⚡ 第二章：AI 响应缓存（4:00 - 9:00）

**旁白：**

> 性能优化的**最高杠杆**：相同问题，直接返回缓存，延迟从 3 秒变成 5 毫秒——600 倍提速！

### 2.1 语义缓存的原理

**画面**：图示，两条相似问题的向量接近

**旁白：**

> 普通缓存只能匹配完全相同的字符串。语义缓存用**向量相似度**判断——"今天天气怎么样" 和 "现在天气如何" 会命中同一条缓存。

### 2.2 配置语义缓存

**画面**：编辑 `~/.openclaw/config.yaml`

```yaml
# ~/.openclaw/config.yaml

cache:
  enabled: true
  backend: redis           # 推荐 Redis，也支持 memory / sqlite
  ttl: 3600                # 缓存有效期（秒）
  
  semantic:
    enabled: true
    threshold: 0.92        # 相似度阈值，0.9-0.95 效果最佳
    embedding_model: text-embedding-3-small   # 嵌入模型
    max_cached_tokens: 500  # 超过此 token 数的回复不缓存（避免缓存过期内容）
    
  # 针对不同场景设置不同策略
  rules:
    - pattern: "^/help"      # /help 命令永久缓存
      ttl: 86400
    - pattern: "天气|weather" # 天气类问题短缓存
      ttl: 1800
    - pattern: ".*"          # 默认策略
      ttl: 3600
```

**旁白（重点提示）：**

> 有几类问题**不应该缓存**：
> - 包含个人信息的（"我的订单状态"）
> - 实时数据类（股价、天气）
> - 需要上下文的多轮对话
>
> 可以用 `exclude_patterns` 排除这些场景。

```yaml
  exclude_patterns:
    - "我的.*"      # 个人化查询
    - "订单|余额|账户"  # 业务数据
    - "帮我写|生成|创作"  # 创作类（结果应该有多样性）
```

### 2.3 缓存效果验证

```bash
# 查看缓存命中率（目标：> 30%）
openclaw cache stats

# 输出示例
# Total requests:    1,247
# Cache hits:          412  (33.0%)  ← 命中率 33%
# Semantic hits:       156  (12.5%)
# Exact hits:          256  (20.5%)
# Avg latency (hit):     4ms
# Avg latency (miss):  3240ms
# Estimated cost saved: $0.82 today
```

**旁白（效果总结）：**

> 33% 的命中率，今天省了 0.82 美元 API 费用，而且这 412 个请求的延迟是 4 毫秒。这才叫**真正的性能优化**。

## 🔗 第三章：HTTP 连接池（9:00 - 13:00）

**旁白：**

> 每次 AI 请求都新建 TCP 连接，光握手就要 50-200ms。连接池让我们**复用已建立的连接**，省掉这笔开销。

### 3.1 问题演示

**画面**：Wireshark 抓包，看到频繁的 TCP SYN/SYN-ACK

```bash
# 查看当前连接状态（没有连接池）
openclaw debug connections

# 输出（连接池关闭时）
# Active connections to api.openai.com: 1
# New connections created (last 60s): 847
# Connection reuse rate: 0.1%  ← 几乎不复用！
```

### 3.2 启用连接池

```yaml
# ~/.openclaw/config.yaml

http:
  connection_pool:
    enabled: true
    max_connections: 50        # 总连接池大小
    max_per_host: 20           # 对同一主机的最大连接数
    keep_alive: true
    keep_alive_timeout: 30000  # 30 秒
    connection_timeout: 5000   # 5 秒连接超时
    idle_timeout: 60000        # 60 秒空闲超时
    
  # 针对 AI 提供商优化
  providers:
    openai:
      base_url: https://api.openai.com
      pool_size: 20
      timeout: 30000
    anthropic:
      base_url: https://api.anthropic.com
      pool_size: 10
      timeout: 60000           # Claude 较慢，给更长超时
```

### 3.3 效果对比

```bash
# 启用连接池后
openclaw debug connections

# 输出（连接池开启后）
# Active connections to api.openai.com: 12
# New connections created (last 60s): 23
# Connection reuse rate: 97.3%  ← 复用率 97%！
# Connection setup time saved: ~140ms per request
```

**旁白（数据解读）：**

> 连接复用率从 0.1% 升到 97%，每个请求省掉约 140ms 的 TCP 握手时间。对于批量处理场景，这个优化非常显著。

### 3.4 DNS 缓存（小技巧）

```yaml
http:
  dns_cache:
    enabled: true
    ttl: 300   # DNS 结果缓存 5 分钟
```

**旁白：**

> 别小看 DNS 解析，`api.openai.com` 的 DNS 查询本身就要 20-50ms。开启 DNS 缓存，又省一刀。

## 📬 第四章：异步消息队列（13:00 - 17:30）

**旁白：**

> 前面两章优化的是**单次请求**的速度。这章解决另一个问题：**高并发下不崩溃**。

### 4.1 问题：同步处理的风险

**画面**：压测工具，模拟 100 个用户同时发消息，错误率飙升

**旁白：**

> 没有队列时，100 个用户同时发消息 = 100 个并发 AI 请求。OpenAI 的 RPM 限制（默认 60 次/分钟）立刻触发 429 错误。用户看到的是：机器人毫无响应。

### 4.2 异步队列架构

**画面**：架构图

```
User Message
     │
     ▼
┌─────────────┐    入队 (< 1ms)    ┌──────────────────┐
│  Channel    │ ─────────────────► │   Message Queue  │
│  Adapter   │                    │  (Redis / BullMQ) │
└─────────────┘                    └────────┬─────────┘
                                            │ 出队处理
                                    ┌───────▼───────┐
                                    │   Workers (N)  │
                                    │  • Rate limit  │
                                    │  • Retry       │
                                    │  • Priority    │
                                    └───────┬───────┘
                                            │
                                    ┌───────▼───────┐
                                    │   AI Provider │
                                    └───────────────┘
```

### 4.3 配置异步队列

```yaml
# ~/.openclaw/config.yaml

queue:
  enabled: true
  backend: redis
  redis_url: redis://localhost:6379
  
  workers: 5               # 并发 Worker 数
  concurrency: 3           # 每个 Worker 同时处理的任务数
  
  rate_limit:
    enabled: true
    requests_per_minute: 55  # 略低于 OpenAI 限制，留出余量
    
  retry:
    max_attempts: 3
    backoff: exponential     # 指数退避
    initial_delay: 1000      # 1 秒
    max_delay: 30000         # 最多等 30 秒
    
  priority:
    # 付费用户优先处理
    rules:
      - condition: "user.plan == 'premium'"
        priority: 10
      - condition: "message.type == 'urgent'"
        priority: 8
      - default_priority: 5
        
  # 等待超过 30 秒告知用户
  pending_notification:
    enabled: true
    threshold: 30000
    message: "消息已收到，正在处理中，稍等片刻 ⏳"
```

### 4.4 压测对比演示

**画面**：两个终端并排，左边无队列，右边有队列

```bash
# 压测工具
openclaw load-test --concurrent 100 --duration 60s --message "介绍一下 OpenClaw"
```

**结果对比（旁白逐行解读）：**

```
=== 无队列 ===                    === 有队列 ===
Requests:     6000               Requests:     6000
Success:      3241 (54%)         Success:      5987 (99.8%)
429 Errors:   2759 (46%)         429 Errors:       4 (0.07%)
Avg latency:  3420ms             Avg latency:  3680ms  (+8%)
P99 latency:  8900ms             P99 latency:  9200ms  (+3%)
Throughput:   54 req/s           Throughput:  99.8 req/s
```

**旁白（重点）：**

> 成功率从 54% 升到 99.8%。代价是平均延迟多了 260ms——这是队列本身的开销。这个取舍完全值得：**用户宁愿多等 0.26 秒，也不愿意看到「发送失败」**。

## 🛡️ 第五章：限流与熔断（17:30 - 20:30）

**旁白：**

> 前面都是「让系统更快」，这章是「防止系统垮掉」。

### 5.1 限流配置（防滥用）

```yaml
# ~/.openclaw/config.yaml

rate_limit:
  per_user:
    free_plan:
      requests_per_minute: 5
      requests_per_day: 100
    premium_plan:
      requests_per_minute: 30
      requests_per_day: 1000
      
  per_channel:
    global:
      requests_per_minute: 200
      
  # 超限时的响应策略
  exceeded_response: "您的请求太频繁了，请稍后再试 🙏"
  exceeded_response_en: "Too many requests. Please slow down 🙏"
```

### 5.2 熔断器配置（防雪崩）

```yaml
circuit_breaker:
  enabled: true
  
  # 当错误率 > 50% 时触发熔断
  failure_threshold: 50
  
  # 至少 10 次请求后才统计错误率
  min_requests: 10
  
  # 熔断后等待 30 秒再尝试半开
  open_duration: 30000
  
  # 半开状态下允许 3 个测试请求
  half_open_requests: 3
  
  # 熔断时的降级响应
  fallback:
    message: "AI 服务暂时不可用，我们正在排查中 🔧 请稍后重试"
    # 可选：指定备用模型
    fallback_model: gpt-4o-mini   # 主模型熔断时自动切换到更稳定的小模型
```

**旁白：**

> 我们来看今天所有优化手段叠加后的最终效果：

```
┌─────────────────────────────────────────────────────┐
│              性能调优前后对比                         │
├─────────────────┬──────────────┬───────────────────┤
```
